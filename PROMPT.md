Векторный поиск был бы нужен, если бы в сметах не было кодов и нужно было сопоставлять по текстовому описанию: "Разводка по устройствам и подключение жил кабелей" → найти похожую расценку в ФСНБ. Но раз коды есть — это не нужно.

Единственное место, где может понадобиться нечёткий поиск по ФСНБ — если подрядчик хочет найти расценку вручную, вбивая текст, а не код. Например, вводит "подключение кабелей 10мм" и ищет подходящий ГЭСН. Но это вспомогательная функция, не критичная для MVP. И даже её проще сделать через полнотекстовый поиск (FTS в SQLite/PostgreSQL) или элементарный LIKE, чем поднимать эмбеддинги.

Итого: ФСНБ хранится как обычная таблица с индексом по коду. Никаких векторных баз, никаких эмбеддингов. Лишняя сложность — враг MVP.

    Если код не распознан — unknown, пометить для ручной классификации

Особенности парсинга:

    Строки с "Итого прямые затраты", "ФОТ", "Накладные расходы", "Сметная прибыль", "Всего по позиции" — это служебные строки, НЕ включать в результат
    Каждая позиция сметы состоит из главной строки (с кодом ГЭСН) и вложенных строк ресурсов (ОЗП, машины, материалы). Нам нужны: главная строка как "работа" + вложенные материалы отдельно
    Разделы определяются по строкам типа "Раздел 1. Монтажные работы ВРУ"
    Строка "Всего по позиции" содержит итоговую стоимость позиции — это "потолок"

Модуль 2: classifier.py — Классификация позиций

Вход: список SmetaItem Выход: тот же список с заполненным item_type

Классификация по префиксу кода из самой сметы — чистый Python, без LLM, без внешних справочников.
Все коды уже присутствуют в смете заказчика, ФСНБ как отдельная база не нужна:
python

def classify(code: str) -> str:
    code_upper = code.upper().strip()
    if any(code_upper.startswith(p) for p in ['ГЭСН', 'ГЭСНМ', 'ГЭСНР', 'ГЭСНП']):
        return 'work'
    if code_upper.startswith('ФСБЦ') or code_upper.startswith('ФСБЦ-'):
        return 'material'
    if code_upper.startswith('ФСЭМ'):
        return 'mechanism'
    if code_upper.startswith('ТЦ_') or code_upper.startswith('ТЛ_'):
        return 'equipment'
    # Коды вида "01.7.15.03-0042" — материалы внутри расценок
    if re.match(r'^\d{2}\.\d+\.\d+', code_upper):
        return 'material'
    # Коды вида "91.05.05-015" — механизмы внутри расценок
    if re.match(r'^91\.\d+', code_upper):
        return 'mechanism'
    # Коды вида "1-100-42" — трудозатраты
    if re.match(r'^\d-\d{3}-\d{2,3}', code_upper):
        return 'labor'
    return 'unknown'

Задача 1: Определение структуры прайса

Промпт:

Ты анализируешь Excel-файл прайс-листа строительного поставщика.
Вот первые 20 строк файла (JSON):
{rows_json}

Определи:
1. Номер строки с заголовками (header_row)
2. Номер колонки с наименованием товара (name_col)
3. Номер колонки с ценой (price_col)
4. Номер колонки с единицей измерения (unit_col), если есть
5. Цена с НДС или без (vat_included: true/false)
6. Ставка НДС если указана (vat_rate)

Ответь строго в JSON без markdown:
{"header_row": N, "name_col": N, "price_col": N, "unit_col": N|null, "vat_included": bool, "vat_rate": N}

Задача 2: Маппинг позиций

Промпт (батчами по 20 позиций прайса):

Ты сопоставляешь позиции из прайс-листа поставщика с ведомостью материалов строительной сметы.

Ведомость материалов (что нужно найти):
{materials_json}

Позиции прайса поставщика:
{pricelist_json}

Для каждой позиции ведомости найди наиболее подходящую позицию из прайса.
Учитывай: марку, сечение, размеры, ГОСТ, тип. "ВВГнг(А)-LS 3×1,5" и "Кабель ВВГнг 3*1.5" — это одно и то же.

Ответь строго в JSON без markdown:
[
  {
    "material_index": 0,
    "pricelist_index": 5,
    "confidence": 0.92,
    "reason": "Совпадение марки и сечения кабеля"
  },
  ...
]

Если подходящей позиции нет, укажи "pricelist_index": null, "confidence": 0.

Обработка ответа:

    confidence >= 0.85 → автоматическое сопоставление (зелёный)
    0.50 <= confidence < 0.85 → предложение, требует подтверждения (жёлтый)
    confidence < 0.50 → не найдено (красный)
